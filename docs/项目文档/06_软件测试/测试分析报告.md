## 三、缺陷统计与分析

### 3.1 缺陷修复记录

本项目在开发过程中共发现并修复 6 个缺陷，全部通过飞书项目管理工具进行跟踪管理。

| 缺陷编号 | Bug 描述 | 严重级别 | 状态 | 根因分析 | 修复方案 | 回归验证 |
|----------|----------|----------|------|----------|----------|----------|
| BUG-001 | 登录报 `UnboundLocalError` | 致命 | 已关闭 | `if not user` 缩进错误，导致 session 处理逻辑进入错误分支 | 修正缩进，将 session 处理放入正确的 `if user` 分支内 | pytest 登录用例 TC-LOGIN-02 验证通过 |
| BUG-002 | 手机号验证码验证失败 | 致命 | 已关闭 | `SESSION_COOKIE_SECURE=True` 导致本地 HTTP 环境下 Session 失效 | 开发环境改为 `SESSION_COOKIE_SECURE=False`，生产环境通过反向代理配置 HTTPS | pytest 注册用例 TC-REG-02 验证通过 |
| BUG-003 | 删除学生时外键约束报错 | 严重 | 已关闭 | 先删除父表 `enrollments` 导致子表 `attendances` 外键冲突 | 调换删除顺序：先删 `attendances`，再删 `enrollments`，最后删 `users` | pytest 删除课程用例 TC-DELC-01 验证级联删除通过 |
| BUG-004 | 签到状态变更时课时数不更新 | 一般 | 已关闭 | 仅处理新建签到记录，未处理已有记录的修改场景 | 增加 `old_status` 判断，状态变化时更新 `attendance_count` | pytest 签到用例 TC-ATD-04 验证通过 |
| BUG-005 | `class_number` 计算错误 | 一般 | 已关闭 | `COUNT(*)` 将 NULL 行也统计进去 | 改为 `COUNT(DISTINCT class_date)` | pytest 签到用例 TC-ATD-03 验证通过 |
| BUG-006 | 添加课程区域太占空间 | 轻微 | 已关闭 | 添加课程表单一直展开，页面冗长 | 添加 `collapsible` 折叠组件，默认收起 | 手工验证 UI |

## 六、安全测试分析

### 6.1 SQL 注入测试

| 注入点 | 测试 Payload | 结果 | 防护机制 |
|--------|--------------|------|----------|
| 登录用户名 | `' OR '1'='1` | 失败 | 参数化查询 `SELECT * FROM users WHERE username = ?` |
| 登录用户名 | `admin'--` | 失败 | 参数化查询 |
| 登录用户名 | `' UNION SELECT * FROM users --` | 失败 | 参数化查询 |
| 消息内容 | `'; DROP TABLE users; --` | 失败（作为纯文本存储） | 参数化查询 + 消息内容无 SQL 执行 |
| 报名 JSON | `["1; DROP TABLE courses;"]` | 失败（表未被删除） | 参数化查询 |

**结论**：所有数据库操作均使用参数化查询，SQL 注入防护有效。

### 6.2 XSS 测试

| 攻击向量 | 测试 Payload | 结果 | 防护说明 |
|----------|--------------|------|----------|
| script 标签 | `<script>alert("XSS")</script>` | 作为纯文本显示 | 后端不解析 HTML，前端 Jinja2 默认转义 |
| img onerror | `<img src=x onerror=alert(1)>` | 作为纯文本显示 | 同上 |
| javascript 伪协议 | `<a href="javascript:alert(1)">点击</a>` | 作为纯文本显示 | 同上 |

**结论**：XSS 攻击被有效防护。建议后续版本中后端对消息内容增加 HTML 转义，前端使用 `textContent` 替代 `innerHTML` 渲染消息。

### 6.3 权限控制测试

| 测试场景 | 普通用户访问管理员接口 | 结果 |
|----------|------------------------|------|
| 添加课程 | POST `/admin/add_course` | 302 重定向到登录页 |
| 删除课程 | POST `/admin/delete_course/1` | 302 重定向到登录页 |
| 访问签到页 | GET `/admin/attendance/1` | 302 重定向到登录页 |
| 访问课时管理 | GET `/admin/hours_management` | 302 重定向到登录页 |
| 调整人员 | POST `/move_student` | JSON 返回 `success=False` |
| 移除学生 | POST `/remove_student` | JSON 返回 `success=False` |
| 临时调课 | POST `/temp_adjust` | JSON 返回 `success=False` |
| 添加学生 | POST `/add_student` | JSON 返回 `success=False` |
| 获取联系人 | GET `/api/contacts` | 403 Forbidden |
| 查看他人对话 | GET `/api/conversation/3` | 403 Forbidden |

**结论**：管理员接口严格校验 `session['role'] == 'admin'`，普通用户无法越权访问。

## 八、测试结论

### 8.1 总体评价

经过全面的自动化测试和手工验证，**墨香书院书法报名管理系统**在功能正确性、接口可靠性、安全防护和数据一致性方面表现良好。146 个 pytest 自动化测试用例全部通过，覆盖核心业务流程和安全场景。

### 8.2 优势总结

1. **功能完整**：系统覆盖了需求规格中定义的全部核心功能，包括用户认证、课程管理、报名缴费、签到课时、消息通信等模块。
2. **安全性可靠**：密码采用 PBKDF2-HMAC-SHA256 加密（salt$hash 格式），SQL 注入通过参数化查询有效防护，权限控制严格区分管理员和普通用户，频率限制和 Session 安全机制均已落实。
3. **数据一致性**：报名缴费、人员调整、删除课程等关键操作均使用数据库事务（BEGIN TRANSACTION / COMMIT / ROLLBACK），确保数据一致性。
4. **实时通信稳定**：WebSocket 连接建立迅速（~80ms），消息广播机制正常，房间管理有效。
5. **性能达标**：所有页面和接口响应时间均低于目标阈值，并发测试通过。

### 8.3 改进建议

| 优先级 | 建议内容 | 预期收益 |
|--------|----------|----------|
| 高 | 修复课程详情页 `course=None` 时的模板报错（ISSUE-01） | 提升用户体验，避免 500 错误 |
| 高 | 生产环境迁移至 MySQL 或 PostgreSQL | 支持更高并发，提升数据可靠性 |
| 高 | 接入真实短信服务和支付接口 | 完善核心业务流程 |
| 中 | 引入 Selenium/Playwright 自动化 UI 测试 | 覆盖前端交互场景，减少回归人力 |
| 中 | 补充性能压测（JMeter/Locust） | 获取系统承载上限数据 |
| 低 | 解决 Windows 开发环境 GBK 编码日志乱码 | 提升开发体验 |
| 低 | 增加后端消息内容的 HTML 转义 | 增强 XSS 纵深防御 |
