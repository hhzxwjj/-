## 二、测试目标

1. **功能正确性**：验证用户认证、课程管理、报名缴费、签到课时、消息通信等核心功能是否符合需求规格说明。
2. **接口可靠性**：验证所有 HTTP API 和 WebSocket 事件的入参校验、返回值格式、异常处理。
3. **安全性**：验证密码加密、SQL 注入防护、XSS 防护、权限控制、频率限制等安全机制的有效性。
4. **数据一致性**：验证数据库事务（报名缴费、人员调整、删除课程等）的 ACID 特性。
5. **兼容性**：验证系统在主流浏览器（Chrome、Edge、Firefox）上的正常运行。
6. **性能基线**：验证页面响应时间和并发处理能力满足预期目标。

## 四、测试策略与方法

### 4.1 测试层次

```
┌─────────────────────────────────────────┐
│  第4层：验收测试（Acceptance Testing）   │  ← 用户场景端到端验证
├─────────────────────────────────────────┤
│  第3层：系统测试（System Testing）       │  ← 完整系统功能验证
├─────────────────────────────────────────┤
│  第2层：集成测试（Integration Testing）  │  ← 模块间接口与数据流
├─────────────────────────────────────────┤
│  第1层：单元测试（Unit Testing）         │  ← 函数/工具独立验证
└─────────────────────────────────────────┘
```

### 4.2 测试方法

| 层次 | 方法 | 说明 |
|------|------|------|
| 单元测试 | 白盒测试 | 对 `hash_password`、`verify_password`、`check_captcha`、`validate_phone` 等工具函数进行独立验证 |
| 集成测试 | 灰盒测试 | 使用 Flask 测试客户端模拟 HTTP 请求，验证路由、中间件、数据库交互 |
| 系统测试 | 黑盒测试 | 基于需求规格说明设计用例，不关心内部实现 |
| 安全测试 | 渗透测试思路 | 使用 SQL 注入payload、XSS payload、权限越界请求进行攻击模拟 |
| 性能测试 | 基准测试 | 使用浏览器 DevTools Network 面板和 Python 脚本测量响应时间 |

### 4.3 测试用例设计方法

1. **等价类划分**：手机号格式（合法/非法）、密码长度（≥6/<6）、课程人数（未满/已满）
2. **边界值分析**：课程最大学生数为 0、验证码 300 秒过期边界
3. **场景法**：正常报名流程、重复报名异常流、信息不完整强制完善资料流
4. **错误推测法**：基于历史 Bug（UnboundLocalError、外键约束、Session 失效）设计回归用例
5. **状态转换**：签到状态（absent → present → absent）的 attendance_count 变化

## 六、测试准入与准出标准

### 6.1 准入条件（Entry Criteria）

1. 全部功能模块开发完成，开发人员已通过自测。
2. 数据库初始化脚本（`init_db()`）可正常运行，表结构完整。
3. 测试环境搭建完毕，pytest 可正常执行。
4. 代码已提交 Git 仓库，版本可追踪。

### 6.2 准出条件（Exit Criteria）

1. 严重（Critical）和高危（High）缺陷修复率 100%。
2. 中低优先级缺陷修复率 ≥ 80%，剩余缺陷已评估风险并接受。
3. 测试用例执行率 100%，通过率 ≥ 95%。
4. 安全测试无高危漏洞未修复。
5. 性能指标全部达标（详见测试分析报告）。
