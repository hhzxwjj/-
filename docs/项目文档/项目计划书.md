# 书法培训班管理系统项目计划书

## 1. 系统概述

### 1.1 项目背景

随着书法培训班规模的扩大，传统的手工管理方式已无法满足日常运营需求。为了提高管理效率，减少人工操作，提升服务质量，特开发书法培训班管理系统。

### 1.2 项目目标

开发一个基于Web的书法培训班管理系统，实现学员管理、课程管理、签到管理、消息通知等功能，为培训班提供全面的信息化管理解决方案。

### 1.3 系统功能

- **用户管理**：注册登录、个人信息管理、密码修改、角色管理
- **课程管理**：课程列表、课程详情、课程报名、缴费流程
- **签到管理**：管理员签到、课时统计、临时调课、课时分析
- **消息通知**：消息中心、实时通知、管理员消息
- **管理员功能**：课程管理、学生管理、数据统计

### 1.4 技术栈

- **后端**：Python + Flask + SQLite
- **前端**：HTML5 + CSS3 + JavaScript + Bootstrap
- **实时通信**：WebSocket (Flask-SocketIO)
- **验证码**：PIL库生成图形验证码

## 11. 风险管理

### 11.1 风险识别

| 风险 | 描述 | 可能性 | 影响程度 |
|------|------|----------|----------|
| 需求变更 | 培训班业务流程调整，导致页面或接口需要修改 | 中 | 高 |
| 权限风险 | 普通用户、管理员、WebSocket 房间权限控制不够细致 | 中 | 高 |
| 安全风险 | 默认管理员弱口令、开发环境验证码直出、调试信息泄露 | 高 | 高 |
| 数据风险 | SQLite 单机部署下并发能力有限，可能出现数据一致性问题 | 中 | 中 |
| 时间风险 | 测试、修复和文档同步耗时超过预期 | 中 | 中 |
| 兼容风险 | 多进程/多实例部署后，内存限流与会话行为不一致 | 中 | 中 |
| 人员风险 | 关键开发人员变动，影响后续维护和交付 | 低 | 中 |

### 11.2 风险应对策略

| 风险 | 应对策略 |
|------|----------|
| 需求变更 | 建立需求变更管理流程，明确变更范围、负责人和回归测试项 |
| 权限风险 | 为路由和 WebSocket 事件增加统一鉴权检查，按角色限制可访问资源 |
| 安全风险 | 生产环境关闭 debug，替换默认口令，接入真实短信服务并限制验证码频率 |
| 数据风险 | 完善事务边界和异常回滚，必要时迁移到更高并发的数据库方案 |
| 时间风险 | 将测试、修复、文档更新纳入里程碑，预留安全加固时间 |
| 兼容风险 | 若部署到多实例环境，使用 Redis/集中式会话替代内存级限流与临时状态 |
| 人员风险 | 保留关键模块说明、部署说明和故障处理手册，降低人员流动影响 |

### 11.3 上线前检查清单

- 确认生产环境关闭 `debug=True`
- 修改默认管理员密码，并建议首次登录强制改密
- 确认验证码接口不再直接回显真实验证码
- 检查所有管理接口是否已做角色校验
- 检查 WebSocket 房间加入/广播是否限制到合法用户
- 检查数据库是否已完成初始化与备份
- 检查日志中是否包含敏感信息（验证码、密码、会话内容）
- 检查限流/风控方案是否适配部署方式

## 12. 支持条件

### 12.1 技术支持

- **开发团队**：负责缺陷修复、版本迭代和紧急响应
- **运维支持**：负责服务器、数据库和日志监控
- **文档支持**：提供部署说明、使用手册和常见问题说明
- **培训支持**：对管理员、教师和学员进行分角色培训

### 12.2 资源支持

- **硬件资源**：建议使用稳定的应用服务器、数据库备份空间和可用网络环境
- **软件资源**：Python、Flask、SQLite、Flask-SocketIO、Pillow 及基础调试工具
- **人力资源**：至少保证后端、前端、测试和运维相关人员可持续支持
- **安全资源**：建议具备 HTTPS 证书、日志审计工具和数据库备份机制

### 12.3 管理支持

- **管理层支持**：明确系统上线范围、审批流程和责任边界
- **用户支持**：管理员、教师和家长需配合测试和验收
- **沟通机制**：建立问题反馈、缺陷跟踪和版本发布机制
- **维护支持**：定期检查安全配置、备份状态和运行日志
